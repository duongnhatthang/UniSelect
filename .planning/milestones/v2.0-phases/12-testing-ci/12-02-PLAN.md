Create `.github/workflows/ci.yml` following the existing workflow conventions in the repo.

The workflow must:
1. Trigger on `pull_request` (all branches, all activity types -- default behavior)
2. Use `ubuntu-latest` runner
3. Use `actions/checkout@v4`
4. Use `actions/setup-node@v4` with `node-version: '20'` and `cache: 'npm'`
5. Run `npm ci` to install dependencies
6. Run `npm test` to execute the test suite
7. Run `npm run build` with env vars for Supabase

For the build step, provide env vars with fallback placeholders so the build does not fail on fork PRs without secrets:
```yaml
env:
  NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL || 'https://placeholder.supabase.co' }}
  NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY || 'placeholder' }}
```

The `generate-static` script in the build command reads from Supabase. If Supabase is unreachable, it may fail. The static JSON fallback files should already exist in the repo from prior deployments. If this is a concern, add `continue-on-error: true` ONLY to the build step, and add a comment explaining why. However, prefer letting it fail naturally -- if the build breaks on a PR, that is useful signal.

Do NOT trigger on `push` -- the requirement says "opening a pull request triggers CI".

**TEST-03: Remove dead `src/` directory**

**TEST-04: Add `public/sw.js` to `.gitignore` and untrack it**
`public/sw.js` is a Serwist build artifact that is currently tracked by git. Two steps required:
1. Add `public/sw.js` to `.gitignore` -- add it in the "# next.js" section since it is a build artifact, or create a new "# serwist" comment section.

IMPORTANT: Both operations must happen together. Adding to `.gitignore` alone does NOT untrack already-tracked files.

After both operations, verify:
- `git ls-files src/` returns empty
- `git ls-files public/sw.js` returns empty
