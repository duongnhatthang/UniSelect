Create `scripts/check-staleness.ts` as a standalone tsx script.

Logic:
1. Read `STALENESS_DAYS` from env (default: 3 for July peak, 7 for off-season). Accept as `STALENESS_DAYS` env var — if not set, default to 7.
2. Query: for each university_id in the `universities` table, find the most recent `scrape_runs` row where `status IN ('ok', 'flagged')`. Use a subquery or CTE with `max(run_at)` grouped by university_id.
3. A university is stale if: no successful run exists, OR most recent successful run's `run_at` is older than `now() - STALENESS_DAYS days`.
4. Print results to stdout:
   - "Checked N universities. M are stale."
   - For each stale university: `  [STALE] {university_id} — last ok run: {date} or never`
5. If M > 0: `process.exit(1)` — GitHub Actions marks the step as failed, which triggers notifications.
6. If M === 0: `process.exit(0)` — clean exit, workflow succeeds.

Use drizzle-orm with the existing `db` import from `../lib/db` and tables from `../lib/db/schema`.
Use `sql` template tag for the `now() - interval` expression:
```typescript
import { sql, max, eq, inArray } from 'drizzle-orm';
// Recent runs subquery:
const recentRuns = await db
  .select({
    university_id: scrapeRuns.university_id,
    last_ok: max(scrapeRuns.run_at),
  })
  .from(scrapeRuns)
  .where(inArray(scrapeRuns.status, ['ok', 'flagged']))
  .groupBy(scrapeRuns.university_id);
```
Then compare `last_ok` to `Date.now() - STALENESS_DAYS * 86400000` in JS.

Close the DB connection after the check (call `sql.end()` if accessible, or let process.exit handle it).

Do NOT use the @/ alias — use relative imports: `../lib/db`, `../lib/db/schema`.

Create `.github/workflows/staleness-alert.yml`:

```yaml
name: Scrape staleness alert

on:
  schedule:
    - cron: '0 8 * * *'   # Daily at 08:00 UTC — after overnight scrape completes
  workflow_dispatch:        # Manual trigger for ad-hoc checks

jobs:
  check-staleness:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - name: Check scrape staleness
        run: npx tsx scripts/check-staleness.ts
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          STALENESS_DAYS: '3'   # Tighter window — alert if any university missed for 3+ days
```

Key points:
- `STALENESS_DAYS: '3'` hardcoded in the workflow (tight window for production alerting).
- Runs at 08:00 UTC daily — after the 01:00 UTC scrape run completes with buffer.
- `workflow_dispatch` for manual checks.
- When the script exits with code 1, GitHub marks the job as failed and sends email notifications to repo watchers — no additional alerting service needed (free tier constraint).
- Do NOT add `continue-on-error: true` — the failure must propagate so GitHub sends alerts.
